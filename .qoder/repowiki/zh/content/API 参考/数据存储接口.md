## 简介
本文件面向 ImgPrompt 扩展的数据存储接口，系统性梳理本地存储的数据结构设计、键名规范、序列化与反序列化处理、访问模式与并发控制、清理与归档策略、迁移与升级方案，并提供可直接定位到源码的示例路径与性能优化建议。目标是帮助开发者与使用者准确理解与高效维护本地数据层。

## 项目结构与数据存储概览
- 存储区域：扩展使用 Chrome Extension Storage API 的 local 区域进行持久化，键名统一以常量形式集中定义于配置模块中，便于维护与一致性校验。
- 关键脚本：
  - background.js：服务工作线程，负责安装初始化、设置默认值、历史记录的增删查、客户端 ID 生成、分析事件上报等。
  - content.js：内容脚本，负责与页面交互、面板状态管理、监听存储变更、发送消息给后台等。
  - options.js：选项页脚本，负责设置表单、自定义模板、历史记录展示与操作、自动保存与通知等。
  - config.js：共享配置，包含默认设置、UI 文案、错误码、分析配置键名等。
  - manifest.json：声明权限与能力，明确 storage 权限。
  - options.html：选项页界面，承载设置与历史记录展示。
- 数据流：设置项与历史记录通过 chrome.storage.local 在后台、内容脚本、选项页之间同步；后台负责写入与清理，前端负责读取与展示。

## 核心数据模型
- 客户端 ID（client_id）
  - 用途：唯一标识用户设备，用于分析事件去重与关联。
  - 键名：CLIENT_ID_KEY
  - 写入时机：首次访问时生成并写入本地存储。
  - 示例路径：[ensureClientId:330-341](file://background.js#L330-L341)

- 设置配置（settings）
  - 用途：扩展运行所需的核心参数，如 API 地址、模型、密钥、语言、最大图像边长、是否启用悬浮按钮等。
  - 默认值：DEFAULT_SETTINGS
  - 读取合并：启动时从本地存储读取，缺失字段用默认值补齐。
  - 示例路径：[loadSettings:322-328](file://background.js#L322-L328)、[DEFAULT_SETTINGS:5-20](file://config.js#L5-L20)

- 历史记录（promptHistory）
  - 用途：保存每次生成的提示词结果、触发来源、时间戳、图片数据等。
  - 结构要点：数组，每条记录包含 id、timestamp、prompts（zh/en）、srcUrl、imageDataUrl、pageUrl、model、trigger 等。
  - 长度限制：MAX_HISTORY_ITEMS，默认 50。
  - 示例路径：[saveToHistory:412-430](file://background.js#L412-L430)、[getHistory:432-440](file://background.js#L432-L440)、[deleteHistoryItem:442-453](file://background.js#L442-L453)、[clearHistory:455-463](file://background.js#L455-L463)

- 自定义模板（customTemplates）
  - 用途：用户自定义的 userPrompt 模板集合。
  - 结构要点：对象映射，键为模板 id，值含 title、prompt。
  - 示例路径：[options.js 中的自定义模板读写:119-149](file://options.js#L119-L149)、[options.js 初始化渲染:182-216](file://options.js#L182-L216)

- 分析配置键（analyticsConfig）
  - 用途：控制是否允许分析事件上报。
  - 键名：ANALYTICS_CONFIG_KEY
  - 示例路径：[safeTrackAnalyticsEvent:404-410](file://background.js#L404-L410)、[config.js:249-251](file://config.js#L249-L251)

## 存储键名规范与命名约定
- 常量集中定义：所有键名均以常量形式在 config.js 中集中声明，避免硬编码散落各处。
- 命名风格：
  - 小驼峰：如 clientId、analyticsConfig、maxImageEdge。
  - 全大写：如 CLIENT_ID_KEY、ANALYTICS_CONFIG_KEY、HISTORY_KEY。
- 建议遵循：
  - 语义清晰：键名应直观表达用途（如 apiEndpoint、hoverButtonEnabled）。
  - 前缀分组：按功能域分组（如 settings_、history_），便于检索与维护。
  - 版本后缀：涉及迁移时可考虑版本号后缀（如 settings_v2），便于灰度与回滚。

## 并发控制与事务处理
- 浏览器存储并发：
  - chrome.storage.local 提供原子性写入，但跨脚本并发仍需谨慎。建议：
    - 单一入口写入：将写入集中在后台或选项页，避免多处同时写入。
    - 批量写入：一次 set 多个键，减少多次写入带来的竞争。
    - 读写顺序：先读后写，避免竞态条件。
- 请求级并发：
  - activeRequests Map 记录进行中的生成请求，支持取消与去重。
  - 示例路径：[activeRequests 管理:17-132](file://background.js#L17-L132)
- UI 并发：
  - content.js 通过 isGenerating、activeRequestId 等状态变量避免重复生成与状态错乱。
  - 示例路径：[content.js 状态变量:36-54](file://content.js#L36-L54)

## 数据迁移与版本升级
- 默认设置注入：
  - 安装或更新时，若本地存储缺少某些键，则以 DEFAULT_SETTINGS 注入，确保新字段可用。
  - 示例路径：[onInstalled 注入默认设置:33-42](file://background.js#L33-L42)
- 键名演进：
  - 新增键名时，优先在 DEFAULT_SETTINGS 中添加默认值，并在 onInstalled 中做补丁。
  - 旧键名废弃时，保留兼容逻辑并在未来版本移除。
- 版本号与升级事件：
  - 通过 manifest.version 识别版本，安装/更新事件中上报分析事件，便于统计升级效果。
  - 示例路径：[安装/更新事件上报:45-56](file://background.js#L45-L56)
- 迁移策略建议：
  - 采用“渐进迁移”：先在 DEFAULT_SETTINGS 中加入新字段，再在后续版本中逐步清理旧结构。
  - 为重要结构引入版本号前缀（如 settings_v2），便于灰度与回滚。
